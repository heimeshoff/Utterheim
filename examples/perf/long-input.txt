Mockingbird is a local-first Windows 11 tray application that gives Claude Code a voice. It synthesizes spoken audio from text, on-device, with no internet, no per-character billing, and no LLM-provider dependency. Its single primary consumer is Claude Code: short notification cues when a task finishes or input is required, and longer read-aloud passes so the user can move around the house with a headset instead of staring at a terminal. Voice diversity is the key feature; different voices route to different parallel Claude sessions so the user can tell sessions apart by ear before they even hear the words.

Mockingbird is the dedicated text-to-speech sibling to WhisperHeim, which handles dictation and call transcription. The two apps share UI technology and design language but are deployed independently. Mockingbird replaces the text-to-speech page that was previously planned for WhisperHeim, since text-to-speech gets pulled out into its own focused app.

The user is a single developer running Claude Code across multiple parallel terminals on a Windows 11 workstation. The user is German but accepts English-only voices for the first version. The user is comfortable with local AI tooling, recording their own voice samples, and capturing system audio from audiobooks, games, podcasts, and video clips. The user triggers speech via Claude Code hooks and via the tray interface directly. There are no other users. There is no multi-tenancy, no authentication beyond local-machine trust, and no telemetry.

When running multiple Claude Code sessions in parallel terminals, the user has no peripheral signal to know which session just finished a task, which session is blocked waiting for input, or what the latest session output actually says, if their attention is elsewhere. Polling terminals visually breaks flow. Generic ding sounds do not distinguish sessions. Cloud text-to-speech services solve the speech part but cost money, leak text to a provider, fail offline, and do not let the user clone arbitrary voices from the audio sources they already have. Mockingbird closes the loop. Every Claude session can speak in a distinguishable voice, locally and instantly enough, with the user owning every voice profile.

The first version is worth shipping when a Claude Code hook can call mockingbird with text and a voice, and audio plays through the user's selected output device within about two seconds. Concurrent calls from multiple sessions queue cleanly, with no overlap and no drops without intent. The user can stop playback instantly with a global hotkey when they want to respond. The user can clone a new voice from a microphone recording or a system-audio capture in under a minute, and use it on the next request. The voice library holds at least five to ten voices reliably, and the architecture clearly accommodates growing to twenty or fifty. The tray application launches, runs unattended in the system tray, and feels like a first-party Windows app, with Mica backdrop, Fluent controls, Segoe UI Variable, and the WhisperHeim aesthetic.

Several capabilities are explicitly out of scope for the first version. Multilingual voices are not in scope; pocket text-to-speech is English-only at launch, and while the architecture should not foreclose a future multi-engine path, no second engine ships in the first version. Reading articles, portable documents, or arbitrary documents is out of scope; read-aloud serves Claude Code output specifically. Generating audio for personal projects, such as narration, podcasts, or game characters, may come later but is not in the first version. Multi-user and multi-machine deployment is out of scope; the tool is strictly local to one workstation. Cloud synchronization of voice profiles is out of scope. Voice profiles live on the local filesystem, and if the user wants synchronization, they can put the folder under cloud storage themselves; mockingbird does not orchestrate it.

Full audio editing is also out of scope, with no clipping, denoising, or mixing interface, just enough capture controls to get a clean ten-second sample. Graphics processing acceleration is out of scope; pocket text-to-speech is processor-only by design, and the first version does not build acceleration code paths. Dictation, transcription, and any speech-to-text feature is out of scope; that is WhisperHeim's job. Authentication and remote-access protection are out of scope; the wire surface is local-only with no token scheme in the first version. A voice marketplace, sharing, or any social feature is out of scope; this is a personal tool only.

In priority order, when in doubt, optimize for privacy, since text never leaves the machine; for offline capability, with no internet dependency at any layer; for independence from any particular language model provider, so the tool works regardless of which model the user runs; for zero recurring cost, with no subscriptions and no per-character billing; and for letting the user own every voice, so they can clone any voice from any audio source they already have. Latency is a secondary concern. Two seconds end-to-end is acceptable. Quality matters but is bounded by the English-only model in the first version.

A handful of open questions are worth revisiting during architecture. The Python and C-sharp integration shape is open, since pocket text-to-speech is a Python library and the host is dot net on Windows; a sidecar process or a bundled Python runtime are candidates for the architect step. The transport from Claude to mockingbird is open, since a local web server, a named pipe, or a command line invocation are all viable, and the architect picks one. The semantics of the stop signal are open, since double tapping the left control key could drain the entire queue or only stop the current utterance; we keep that ambiguous in the vision and revisit on first usage. The decoupling of concurrent playback is open, since read-aloud and notification cues might want different lanes; whether a finished-task cue can barge to the front of the queue or whether the queue is strictly first in first out is a decision we will make once the user has run a few sessions in anger. The organization of the voice library is open, since twenty to fifty voices on a flat list will work but probably want at least search and filter; tags, categories, or favourites are options to revisit once we have around fifteen voices. The upgrade path to multilingual is open; the architecture should leave room for a second engine, and a voice profile should know which engine produced it. Finally, the question of reuse versus duplication with WhisperHeim is open, since its audio capture, hotkey, model download, and tray infrastructure are reusable, but the exact form of reuse is out of scope for the vision and is resolved during foundation.
